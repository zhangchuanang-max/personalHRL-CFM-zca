你的代码不是从零写的，而是基于一篇 2024 年的论文改造的：

## 二、你的开题报告要求了什么

开题报告提出了 **3 个创新点**，简单理解：

### 创新点 1：双层指挥系统

**大白话**：现在网络是"一把抓"——所有任务同等对待。你要改成"先定大方向、再做具体分配"。

```
现在的做法（单层）：
  指挥官 → 直接看所有任务 → 选一个
  
你要做的（双层）：
  上层指挥官 → 先决定"现在重点搞哪个区域" → 画个圈
  下层执行员 → 只在圈里选具体任务
```

比如：上层说"现在优先处理 Zone C 的紧急任务"，下层就只在 Zone C 的任务里选，不看别的区域。

### 创新点 2：让机器人"心有灵犀"

**大白话**：机器人 A 选了一个需要两个人的任务，机器人 B 应该能"感觉到"A 已经去了，自己主动去帮忙。

```
现在的做法：
  A 选了任务 X → B 也看到了任务 X → 但 B 不知道 A 选了 → B 可能去撞车
  
你要做的：
  A 选了任务 X → 系统立刻告诉 B"任务 X 还差一个人" → B 看到后主动去补位
```

其实代码里**已经有了大部分**这个功能（原论文自带），但缺一个关键环节（后面会讲）。

### 创新点 3：严格的安全检查

**大白话**：在机器人做决定之前，先用"安全规则"把不该去的任务全部划掉。

```
你要做的安全规则：
  - Zone A 只让 Type 1 进（太窄了，大块头进不去）
  - Zone B 只让 Type 2 和 Type 3 进（有辐射，小机器人扛不住）
  - Zone C 谁都能进
  - 已经满员的任务不让人再去了
```

### 改造 3：CFM 的"循环回来"机制

**难度**：⭐⭐⭐（改 while 循环逻辑）

**这是什么**：论文里说，当一个机器人发现自己"啥也干不了"的时候，不是直接跳过，而是让别的机器人先做决定，等它们做完后，再回来给这个机器人一次机会。这样有可能别的机器人"腾出"了新任务。

**现在的问题**（`worker.py` run_episode 方法）：

```python
while release_agents[0] or release_agents[1]:
    agent_id = release_agents[0].pop(0) if release_agents[0] else release_agents[1].pop(0)
    ...
    block_flag = mask[0, 1:].all().item()
    if block_flag and ...:
        agent['no_choice'] = block_flag
        continue   # ← 问题在这：跳过之后就再也不回来了！
```

**应该改成**：

```python
# 新增一个"等待队列"
deferred_agents = []

while release_agents[0] or release_agents[1] or deferred_agents:
    # 先处理正常释放的机器人，处理完了再处理等待队列里的
    if release_agents[0]:
        agent_id = release_agents[0].pop(0)
    elif release_agents[1]:
        agent_id = release_agents[1].pop(0)
    else:
        agent_id = deferred_agents.pop(0)  # ← 循环回来！
    ...
    if block_flag and ...:
        deferred_agents.append(agent_id)   # ← 不再直接跳过，而是加入等待队列
        continue
    # 正常做决策...
```

**给 AI 的提示词**：

```
请修改 worker.py 中 run_episode() 方法的决策循环，实现论文中的 CFM cycle-back 机制。

论文中的 CFM 机制：
当某个机器人发现自己无法对任何任务有贡献时（mask 全为 True），不要直接跳过它。
而是：先让它等待，让其他机器人先做决策 → 等其他人都做完了 → 循环回来再给它一次机会。
重复这个过程直到它能做出选择，或者这一步所有机器人都处理完了。

当前代码的问题：
```python
if block_flag and not np.all(...):
    agent['no_choice'] = block_flag
    continue   # ← 跳过之后不再回来
```

修改方案：

1. 在 while 循环之前，新增一个 list：deferred_agents = []

2. 修改 while 循环的条件：
   while release_agents[0] or release_agents[1] or deferred_agents:

3. 修改 agent 取出顺序：
   - 优先从 release_agents[0] 取
   - 其次从 release_agents[1] 取
   - 最后从 deferred_agents 取（这就是 cycle back）

4. 修改 block_flag 的处理：
   - 不再直接 continue
   - 改为 deferred_agents.append(agent_id)
   - 设置一个计数器防止无限循环（同一个 agent 最多 cycle back 3 次）

5. 注意：
   - 每次 cycle back 回来时，需要重新调用 agent_observe 获取最新 mask
     （因为其他 agent 的决策已经改变了环境状态）
   - 推理时不启用 CFM（在 parameters.py 加一个 USE_CFM 开关）
```

### 改造 5：Deadline 可达性预判

**难度**：⭐⭐（新增一个方法）

**这是什么**：机器人在决定去哪个任务之前，先算一下"我赶过去来不来得及"。如果来不及在 deadline 之前到，就不选这个任务。

**给 AI 的提示词**：

```
请在 task_env.py 中实现 Deadline 可达性预判。

问题：当前代码中，任务超时是在执行过程中事后判定的。
机器人在选择任务时不知道"这个任务已经来不及了"，
可能白跑一趟。

实现：

1. 新增方法 get_deadline_feasible_mask(self, agent_id)：
   - 返回长度为 tasks_num 的 bool 数组
   - 对于每个未完成任务：
     - 计算机器人到任务位置的行驶时间 = 距离 / 速度
     - 最乐观完成时间 = 当前时间 + 行驶时间 + 任务执行时间 task['time']
     - 如果最乐观完成时间 > deadline → mask 为 True（来不及了，别去了）
   - 注意：这里的"最乐观"假设所有队友同时到达，实际上会更慢，
     所以这是一个保守估计，宁可错过也不白跑

2. 在 agent_observe() 中合并此掩码：
   deadline_mask = self.get_deadline_feasible_mask(agent_id)
   mask = np.logical_or(mask, deadline_mask)

3. （可选）在 parameters.py 中加一个开关 ENABLE_DEADLINE_PRUNING = True
```

### 改造 8：双层架构（最复杂，建议最后做）

**难度**：⭐⭐⭐⭐⭐（需要改网络结构 + 训练流程）

**这是什么**：目前网络是"一层到底"——直接看所有任务然后选一个。要改成"先选区域、再选任务"的两层结构。

**这个改造为什么最难**：
- 需要改神经网络的结构（`attention.py`）
- 可能需要改训练流程（`driver.py`、`worker.py`）
- 改完后需要重新训练模型

**建议**：先完成改造 1-7，确认训练效果有提升后，再考虑这个。

**给 AI 的提示词**（供后续使用）：

```
请在 attention.py 中为 AttentionNet 添加上层阶段控制器。

这是最大的改造项。目标：在现有单层网络之上，新增一个"上层"模块，
让网络先决定"关注哪个区域"，再在选定区域内做具体任务分配。

方案（先做最简版本，不学可微分的上层策略）：

1. 在 AttentionNet.__init__() 中新增：
   self.zone_classifier = nn.Linear(embedding_dim * 3, 3)  # 3个zone

2. 在 AttentionNet.forward() 中：
   - 用融合后的特征通过 zone_classifier，输出 3 个 zone 的分数
   - Softmax 得到 zone 优先级
   - 根据每个任务的 x 坐标（需要从 task_inputs 中提取），
     将任务按 zone 分组
   - 生成 macro_mask：优先级最高的 zone 的任务不被 mask，其他 zone 的任务被 mask（设为 -inf）
   - 将 macro_mask 与传入的 global_mask 合并

3. 注意：task_inputs 中需要包含任务的 x 坐标信息。
   请先检查 task_env.py 中 get_current_task_status() 输出的特征向量，
   确认 x 坐标在哪个位置，然后在 AttentionNet 中正确提取。

4. 这个改造会改变网络的输入输出维度，改完后必须从头训练。

请先阅读 attention.py 和 task_env.py 中的相关代码，理解当前数据流后，
再给出详细的修改方案让我确认。
```

## 七、怎么让 AI 帮你改代码

### 基本流程

```
1. 复制上面的"给 AI 的提示词"
2. 打开 AI 编程工具（如 Claude、ChatGPT、GitHub Copilot Chat）
3. 先把相关代码文件的内容发给 AI（比如改 task_env.py 就发这个文件）
4. 粘贴提示词
5. 让 AI 输出修改后的代码
6. 复制 AI 的输出，替换原文件中对应的部分
7. 运行 python test.py 或 python quick_view.py 测试
```

### 注意事项

1. **每次只改一项**：不要一次把所有改造都告诉 AI，容易出错
2. **改完先测**：运行 `python quick_view.py` 看看有没有报错
3. **保留备份**：改文件前复制一份原文件，出问题了可以回退
4. **重新训练**：改造 2/3/4/5/6 改了 mask 逻辑后，旧模型的权重可能不兼容，需要重新训练
5. **测试集要重做**：改造 2（Zone 准入）改了环境逻辑后，旧的测试集 .pkl 文件需要重新生成

### 用什么 AI 工具

| 工具 | 适合场景 | 用法 |
|------|---------|------|
| Claude (claude.ai) | 一次性改造，代码量不大 | 把代码贴进去 + 贴提示词 |
| ChatGPT | 同上 | 同上 |
| GitHub Copilot Chat | 在 VS Code 里直接改 | 选中代码 + 右键 + Ask Copilot |
| Cursor | 整个文件级别的改造 | 直接在编辑器里对话 |
